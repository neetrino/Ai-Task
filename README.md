# Bitrix24: текст проекта → эпики и задачи

Интеграция задумана так: вы передаёте в чат текст/аккаунт проекта, ассистент структурирует работу (эпики → задачи), затем создаёт объекты в Bitrix24 **в нужном проекте** (идентификатор группы/проекта задаётся в окружении). Один входящий вебхук на портал; для смены проекта меняется только ID группы (Scrum/проект), URL вебхука не меняется.

## Переменные окружения

| Переменная | Назначение |
|------------|------------|
| `Webhook_URL` | Базовый URL вебхука **с завершающим слэшем**: `https://<портал>/rest/<user>/<secret>/` |
| `Bitrix24_Project_id` | ID **рабочей группы / проекта / Scrum-команды** в Bitrix24 (в REST часто передаётся как `groupId` для эпиков или `GROUP_ID` / `groupId` в полях задачи). Это не «ID вебхука». |

Рекомендуется держать шаблон без секретов в `.env.example` и не коммитить `.env`.

Официальная документация по вебхукам: [Локальные интеграции](https://apidocs.bitrix24.ru/local-integrations/local-webhooks.html).

## Права вебхука (scopes)

Скрин с наборами `task`, `tasks`, `tasks_extended`, `tasksmobile` для **создания и списка задач, расширенных полей** в целом **достаточен**, но для вашего сценария (проект, эпики Scrum, привязка к группе) обычно нужно **добавить** права на **рабочие группы / соцсеть** (в списке прав это близко к **«Рабочие группы» / `sonet_group` / соцсеть** — точное имя зависит от языка интерфейса портала).

| Scope (как в API) | Комментарий |
|-------------------|-------------|
| **`task`** | Задачи (базовые методы). Имеет смысл **оставить**. |
| **`tasks`** | Модуль задач (`tasks.task.*` и др.). Имеет смысл **оставить**. |
| **`tasks_extended`** | Расширенные поля, часть сценариев Scrum/кастомных полей. Для «эпики + задачи с полями» **лучше оставить**. |
| **`tasksmobile`** | Мобильный контекст. Для серверных вызовов из чата/скрипта **обычно лишний**; можно **снять**, если после снятия всё работает. |
| **`sonet_group`** (и при необходимости смежные права соцсети) | Для **Scrum-эпиков** (`tasks.api.scrum.epic.*`) и устойчивого доступа к **проекту/группе** — **желательно добавить**. Без этого возможны `insufficient_scope` или «нет доступа к скраму». |

Итого: **лишним для вашей цели чаще всего бывает только `tasksmobile`**. **`task` + `tasks` + `tasks_extended`** — не дублируют друг друга полностью; спорные пары лучше **не отключать**, пока не убедитесь по тестам. Если планируете **только** методы `tasks.task.*` без Scrum-эпиков, всё равно проверьте создание задачи с `GROUP_ID` равным вашему проекту — при ошибке прав **добавьте `sonet_group`**.

Эпики в смысле **Scrum**: методы вида `tasks.api.scrum.epic.add` принимают `groupId` (ваш проект/скрам). См. [документацию эпика](https://apidocs.bitrix24.ru/api-reference/sonet-group/scrum/epic/tasks-api-scrum-epic-add.html).  
Если «эпик» у вас — **просто крупная родительская задача** без модуля Scrum, может хватить `tasks.task.add` и связей между задачами — тогда отдельные права Scrum могут не понадобиться, но **`GROUP_ID` всё равно должен указывать на нужный проект**.

**Важно:** вебхук действует от пользователя, который его создал. Этот пользователь должен быть участником проекта/Scrum и иметь право создавать задачи и эпики.

## Проверка подключения

```bash
# Замените URL на свой вебхук (со слэшем на конце)
curl -sS "https://YOUR_PORTAL.bitrix24.ru/rest/1/SECRET/profile.json"
```

Успешный ответ содержит `result` с полями профиля.

## Инструкция для AI-ассистента

1. **Контекст:** один базовый `Webhook_URL`; целевой проект — `Bitrix24_Project_id` (число, ID группы/проекта в Bitrix24).
2. **Вход пользователя:** сырой текст о проекте (цели, этапы, ограничения). Не выдумывать факты; при неясности — кратко уточнить.
3. **План:** сначала предложить структуру: эпики (крупные блоки) → задачи под каждым эпиком; согласовать или по запросу пользователя сразу перейти к созданию в Bitrix24.
4. **Создание в Bitrix24:**
   - Эпики Scrum: `tasks.api.scrum.epic.add` с `fields.groupId` = `Bitrix24_Project_id` (и имя, описание и т.д.).
   - Задачи: `tasks.task.add` с полями, среди которых для привязки к проекту обычно указывают идентификатор группы (в документации к методу — `GROUP_ID` / `groupId` в `fields`).
5. **Ошибки:** `insufficient_scope` → попросить человека добавить недостающие права в настройках вебхука; `ACCESS_DENIED` / «нет доступа к скраму» → проверить участие пользователя вебхука в проекте и тип проекта (Scrum/не Scrum).
6. **Секреты:** не логировать полный URL вебхука в открытых чатах и репозитории.

Справочник методов: [Задачи](https://apidocs.bitrix24.ru/api-reference/tasks/index.html), [Эпики Scrum](https://apidocs.bitrix24.ru/api-reference/sonet-group/scrum/epic/api-reference/sonet-group/scrum/epic/index.html).

## Файлы

- `.env` — локальные значения (не коммитить).
- `.env.example` — шаблон имён переменных без секретов.
- `.cursor/rules/*.mdc` — правила Cursor для этого репозитория (поток ТЗ → YAML → sync, без секретов в коде).

Репозиторий: [github.com/neetrino/Bitrix24-Task](https://github.com/neetrino/Bitrix24-Task).

## План в YAML → проверка → заливка в Bitrix (вариант 1)

**Чем читать план:** для людей удобнее **YAML** — комментарии, нормальные переносы строк в описаниях, без лишних кавычек. **JSON** пригоден как обмен между системами, но для ручного редактирования хуже. Используйте один файл плана на проект (или копию шаблона).

1. Скопируйте `plans/example.plan.yaml` в свой файл, например `plans/marco.local.yaml` (локальные копии с суффиксом `.local.yaml` в `.gitignore` — не коммитьте чужие ТЗ).
2. Заполните эпики и задачи, **прочитайте файл целиком**, поправьте формулировки.
3. Сухой прогон (ничего не создаёт в портале):

   ```bash
   npm install
   npm run sync -- plans/example.plan.yaml --dry-run
   ```

4. Создание в Bitrix24:

   ```bash
   npm run sync -- plans/example.plan.yaml
   ```

В `.env` должны быть `Webhook_URL` и `Bitrix24_Project_id`. Опционально `Bitrix24_responsible_id` — ID ответственного по умолчанию (если не задан `responsible_id` в YAML).

**Режимы `epic_mode` в плане:**

| Значение | Поведение |
|----------|-----------|
| `scrum` | Эпики через `tasks.api.scrum.epic.add`, задачи с полем `epicId` (нужны права на Scrum/группу). |
| `parent_tasks` | «Эпик» создаётся обычной задачей, подзадачи с `PARENT_ID` — без API эпиков, работает везде, где есть задачи. |

Если при `scrum` Bitrix вернёт ошибку по полю эпика, переключите план на `parent_tasks` и повторите.
